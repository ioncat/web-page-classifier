# ML-классификатор для url-parser — план разработки

**Дата планирования:** 08.03.2026
**Железо:** AMD Ryzen 5 5500 / NVIDIA RTX A4000 (16GB VRAM) / 64GB DDR4
**Контекст:** LLM-разметка (step3) генерирует шумные метки → используем их как обучающие данные для быстрого специализированного классификатора.

---

## Мотивация

Текущая LLM-классификация (step3 / Ollama):

| Проблема | Проявление |
|----------|------------|
| Непоследовательный регистр | `управление командами` / `Управление командами` |
| Дубликаты рус/англ | `ИИ-ассистенты` и `AI-ассистенты` — одно и то же |
| Мусорная корзина | `Оборудование и инструменты` — 20+ несвязанных тем |
| Скорость | ~2–5 с/URL, нужен запущенный Ollama |
| Недетерминированность | один URL → разный тег при повторном запуске |

Цель ML-классификатора:

- **4000 URL за ~2 секунды** (vs ~2–4 часа через Ollama)
- Детерминированный результат
- Без GPU/Ollama на инференсе
- macro-F1 > 0.80

## Фаза 2 — Разметка данных (3–5 дней)

### Шаг 2.1 — LLM-разметка с фиксированной таксономией (`--strict`)

Реализовать `--strict` режим в step3 (уже в беклоге как P1):

```bash
# Модель выбирает ТОЛЬКО из taxonomy.json, не изобретает
python main.py --re-tag --strict --model qwen2.5-coder:7b --batch 10 --workers 4
```

Промпт в `--strict` режиме меняется с:
```
"You may use a suggested category OR invent your own"
```
на:
```
"You MUST choose exactly one category from this list: [список]
Respond with ONLY the category name, exactly as written."
```

### Шаг 2.2 — Экспорт датасета

Новый скрипт `export_dataset.py`:

```bash
python export_dataset.py --output dataset.jsonl
# Формат строки:
# {"text": "Как устроены переменные в Python [SEP] habr.com", "label": "Программирование"}
```

Вход модели = `f"{title} [SEP] {domain}"` — domain даёт важный контекст
(github.com → скорее всего код; habr.com → технический блог).

> **Примечание о разнообразии датасета.** Корпус сформирован из личных закладок
> одного пользователя: ~70% статей — IT/AI/программирование, остальные 30% —
> науч-поп, бизнес, карьера, продуктивность. Это нормально для персонального
> инструмента, но модель будет хуже работать на темах с малым числом примеров
> (финансы, IoT, история). При расширении корпуса стоит целенаправленно добирать
> примеры из редких категорий.

### Шаг 2.3 — Ручная валидация (критически важно)

```bash
# Новый флаг для ручной правки
python main.py --fix-category URL "Правильная категория"
```

Стратегия валидации:
- 20–30 случайных примеров на каждую категорию
- В первую очередь правим категории с низкой очевидностью
- **Цель: ≥50 чистых примеров на категорию** = ~750–1000 проверенных записей

> При 15 категориях и ~4000 URL — у нас уже почти достаточно данных.
> Ручная валидация нужна только для повышения чистоты, не объёма.

**Артефакт:** `dataset.jsonl` (~4000 строк, из них ~1000 верифицированы вручную).

## Фаза 3 — Обучение модели (1 день)

### Стек

```
PyTorch + HuggingFace Transformers + scikit-learn (метрики)
```

### Выбор базовой модели

| Модель | Размер | VRAM | Обучение (RTX A4000) | Инференс CPU |
|--------|--------|------|----------------------|--------------|
| `rubert-tiny2` | 30M | ~1GB | ~30 сек | <1 мс |
| `xlm-roberta-base` | 280M | ~3GB | ~3 мин | ~2 мс |
| `xlm-roberta-large` | 560M | ~6GB | ~8 мин | ~5 мс |

**Выбор: `xlm-roberta-base`** — оптимальный баланс качества и скорости.
Понимает русский и английский, хорошо работает на коротких текстах.

### Параметры обучения (`train_classifier.py`)

```python
model = AutoModelForSequenceClassification.from_pretrained(
    "xlm-roberta-base",
    num_labels=len(taxonomy)   # 15
)

# Вход: f"{title} [SEP] {domain}"  (токенизируется вместе)
# Выход: индекс категории → taxonomy[idx]

batch_size    = 32      # RTX A4000 легко тянет
epochs        = 5       # обычно достаточно
learning_rate = 2e-5    # стандарт для fine-tune BERT
warmup_steps  = 100
weight_decay  = 0.01

train / val / test = 70 / 15 / 15  # стратифицированный split
```

### Метрики оценки

```
accuracy      — общая точность
macro-F1      — важно: классы несбалансированы (Habr → много AI/программирования)
confusion matrix → видим где модель путает (напр. AI vs Data Science)

Цель: macro-F1 > 0.80
```

**Артефакты:**
- `models/url_classifier/` — сохранённая HuggingFace модель (~500MB)
- `models/url_classifier/confusion_matrix.png` — матрица ошибок
- `models/url_classifier/metrics.json` — accuracy, F1 по классам

## Фаза 4 — Интеграция в пайплайн (2 дня)

### Новый `step4.py` — ML-классификатор

```
pipeline:
  step1 (импорт) → step2 (парсинг) → step3 (LLM) ИЛИ step4 (ML)

                                         ↓ confidence < порога
                                      fallback → step3 (LLM)
```

**Логика step4:**
1. Загружает модель из `models/url_classifier/`
2. Классифицирует батчами (CPU: 500 URL/с, GPU: мгновенно)
3. Для каждого URL: category + confidence (softmax score)
4. Если confidence < `--ml-confidence` (по умолчанию 0.70) → fallback на LLM

### Новые флаги в `main.py`

```bash
python main.py --only-ml-classify                    # только step4 (ML)
python main.py --only-ml-classify --ml-confidence 0.8  # строгий порог
python main.py --ml-model models/url_classifier      # путь к модели
```

### Хранение в БД

```sql
-- Добавить колонку ml_confidence в urls
ALTER TABLE urls ADD COLUMN ml_confidence REAL;

-- tagged_by хранит: "NextAgent/Convo:latest" (LLM) или "xlm-roberta-base" (ML)
```

## Фаза 5 — Итерация (непрерывно)

```
Новые URL
    → step4 (ML, мгновенно)
    → confidence < 0.70 → step3 (LLM, медленно)
    → человек правит через --fix-category
    → накопилось 200+ правок → переобучение (3 мин)
    → новая версия модели → деплой
```

**Active learning:** при ревью показываем сначала примеры с наименьшей
confidence — это самые полезные для обучения.

**Цикл переобучения:** раз в месяц или при накоплении ~200 новых верифицированных примеров.

## Долгосрочные направления

### Иерархическая таксономия (L1 → L2 → L3)

Текущая классификация — плоский список L1. Следующий уровень:

```
L1: Искусственный интеллект
  L2: LLM
    L3: RAG, Fine-tuning, Prompting
  L2: Компьютерное зрение
  L2: MLOps
```

Подходы:
- **Каскадные классификаторы** — отдельная модель на каждый уровень (L1 → если "ИИ" → L2-ИИ → ...). Просто, интерпретируемо.
- **Multi-label** — одна модель предсказывает сразу все уровни. Сложнее, но быстрее.

Пререквизит: стабильная L1-таксономия (Фаза 1–3 текущего плана).

### Обогащение признаков (step2)

Сейчас вход модели: `title [SEP] domain`. Планируется:

| Признак | Источник | Ценность |
|---------|----------|----------|
| `og:description` | `<meta property="og:description">` | Часто информативнее title |
| `meta description` | `<meta name="description">` | Краткое резюме страницы |
| `og:title` | `<meta property="og:title">` | Альтернативный заголовок |

Новый вход: `f"{title} [SEP] {description[:200]} [SEP] {domain}"`.

Трейдоff: больше токенов → чуть медленнее LLM, но ML-классификатору только лучше.
